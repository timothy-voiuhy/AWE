# Session Token Entropy

## Overview
Session token entropy refers to the randomness and unpredictability of session identifiers generated by a web application. Low-entropy tokens can be predicted or brute-forced by an attacker, allowing them to hijack another user's session. This vulnerability exists when developers use custom session ID generation routines instead of cryptographically secure random number generators, or when tokens incorporate predictable data like timestamps or user IDs.

## How It Works
- The server generates a session token and assigns it to a user upon login.
- If the token space is small or the generation algorithm is weak, an attacker can enumerate or predict valid tokens.
- Sequential or time-based tokens (e.g., Unix timestamp + user ID) can be guessed using simple arithmetic.
- Tokens with insufficient bit-length (e.g., 32-bit) can be brute-forced within minutes against a live application.
- An attacker captures several tokens, analyzes patterns, then crafts or guesses tokens belonging to other sessions.
- The server accepts the crafted token and grants the attacker access to the victim's session.

## Impact
- Full account takeover of arbitrary users without knowing their credentials.
- Access to sensitive user data, PII, payment information, and private messages.
- Privilege escalation if an attacker can predict an admin session token.
- Mass session hijacking across all active user sessions.
- Persistent unauthorized access without triggering authentication checks.

## Where to Look
- `Set-Cookie` response headers — examine the session cookie value length, character set, and apparent randomness.
- Login responses, registration responses, and password reset flows that issue new session tokens.
- Tokens embedded in URLs (e.g., `?sessionid=...`) or hidden form fields.
- Application source code for custom random number generation (e.g., `rand()`, `Math.random()`, timestamp-based IDs).
- Token values that contain encoded data (Base64-decode them; look for timestamps, user IDs, email addresses).
- Sequential token values by logging in multiple times with test accounts.

## Testing Steps
1. Create two or more test accounts and log into each separately.
2. Capture the session token issued to each account from the `Set-Cookie` header.
3. Record 20–50 tokens by repeatedly logging in and logging out.
4. Decode each token (Base64, hex, URL-decode as applicable) and compare values side by side.
5. Check token length — OWASP recommends at least 128 bits (16 bytes) of randomness.
6. Calculate the Shannon entropy of the token set; low entropy indicates predictability.
7. Look for patterns: incrementing numbers, timestamps, user ID substrings, or repeated segments.
8. Use Burp Suite Sequencer to capture a live feed of tokens and run a statistical analysis.
9. If patterns are found, attempt to construct a token for a known concurrent session and replay it.
10. Verify the forged token is accepted by sending it in a request that returns user-specific data.

```bash
# Capture multiple session tokens via curl
for i in $(seq 1 20); do
  curl -s -c /dev/null -b /dev/null -X POST https://target.com/login \
    -d "username=testuser&password=testpass" -I 2>&1 | grep -i set-cookie
  sleep 0.5
done

# Decode Base64-encoded tokens
echo "dXNlcl9pZD0xMjM0JnRpbWU9MTY4MDAwMDAwMA==" | base64 -d
# Output: user_id=1234&time=1680000000  <-- predictable!

# Analyse token entropy with python
python3 -c "
import base64, math, collections
tokens = [
    'dXNlcl9pZD0xJnRpbWU9MTY4MDAwMDAwMA==',
    'dXNlcl9pZD0yJnRpbWU9MTY4MDAwMDAwMQ==',
]
for t in tokens:
    data = base64.b64decode(t)
    freq = collections.Counter(data)
    entropy = -sum((c/len(data)) * math.log2(c/len(data)) for c in freq.values())
    print(f'Token: {t[:20]}... Entropy: {entropy:.2f} bits/byte')
"

# Brute-force short token space (32-bit token example)
python3 -c "
import requests, itertools, string
charset = string.hexdigits[:16]
for combo in itertools.product(charset, repeat=8):
    token = ''.join(combo)
    r = requests.get('https://target.com/dashboard', cookies={'session': token})
    if r.status_code == 200 and 'Welcome' in r.text:
        print(f'Valid token found: {token}')
        break
"
```

```python
# Token pattern analysis script
import re
from datetime import datetime

tokens = [
    "sess_1680000001_user42",
    "sess_1680000002_user43",
    "sess_1680000003_user44",
]

for t in tokens:
    m = re.match(r'sess_(\d+)_user(\d+)', t)
    if m:
        ts, uid = int(m.group(1)), int(m.group(2))
        dt = datetime.fromtimestamp(ts)
        print(f"Token: {t} | Timestamp: {dt} | UserID: {uid}")
        print("  --> PREDICTABLE: timestamp + sequential user ID")
```

## Burp Suite Tips
- **Sequencer**: Send the login request to Burp Sequencer, set the token location (cookie or response body), then capture 200+ live tokens. Run the FIPS analysis — a token passes if it scores > 23 bits per character for a reasonable-length token.
- **Intruder (Brute-force)**: If tokens appear short or hex-only, use Intruder with a brute-force payload type targeting the cookie header. Set concurrency high to speed up enumeration.
- **Comparer**: Capture tokens from multiple logins and compare them with Burp Comparer to visually spot common prefixes, suffixes, or structural patterns.
- **Decoder**: Always run suspected tokens through Burp Decoder (Base64, URL, hex) to reveal any embedded plaintext information.
- **Repeater**: Manually craft a token based on an identified pattern and replay it to verify successful session hijacking.

## Tools
- **Burp Suite Sequencer** — statistical analysis of session token randomness built into Burp Suite.
- **OWASP ZAP Session Analysis** — ZAP has a built-in spider and session token scanner.
- **Hetty** — lightweight HTTP proxy with session analysis capabilities.
- **Python requests + itertools** — custom scripts for pattern detection and brute-force.
- **CyberChef** — browser-based tool for decoding and analyzing token encoding formats.
- **jwt_tool** — if tokens are JWTs, analyzes signing algorithm, claims, and tampering attacks.
- **TokenBreaker** — specialized tool for session token entropy analysis and prediction.

## Remediation
- Use a cryptographically secure pseudo-random number generator (CSPRNG) — e.g., `secrets.token_hex(32)` in Python, `SecureRandom` in Java, `crypto.randomBytes(32)` in Node.js.
- Ensure session tokens are at least 128 bits (16 bytes) of actual randomness.
- Never incorporate predictable data (timestamps, user IDs, IP addresses) into the token.
- Use the platform's built-in session management framework rather than rolling a custom implementation.
- Rotate session tokens on every privilege change (login, role escalation, sudo actions).
- Enforce a short session expiration window and regenerate tokens on re-authentication.
- Audit session generation code in every release cycle.
